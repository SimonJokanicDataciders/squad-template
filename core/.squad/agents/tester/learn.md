# Tester Agent — Codebase Learning Protocol

When instructed to learn the codebase, execute these steps IN ORDER.
Save your findings to `.copilot/skills/role-qa-core.md` using the skill bundle format.

---

## Step 1: Identify the Testing Stack

```
# Find test files
find . -type f \( -name "*.test.*" -o -name "*.spec.*" -o -name "*Test.cs" -o -name "*_test.go" -o -name "test_*.py" \) -not -path '*/node_modules/*' | head -30

# Check test configuration
ls jest.config.* vitest.config.* pytest.ini .pytest.ini setup.cfg tox.ini phpunit.xml karma.conf.* 2>/dev/null

# Check test commands
cat package.json 2>/dev/null | grep -A5 '"test"'
```

Identify:
- **Test runner:** (Jest, Vitest, pytest, xUnit, go test, JUnit, PHPUnit, etc.)
- **Assertion library:** (built-in, Chai, Should.js, FluentAssertions, etc.)
- **Mocking:** (jest.mock, unittest.mock, Moq, testify, etc.)
- **Integration test setup:** (Testcontainers, supertest, httptest, WebApplicationFactory, etc.)
- **E2E:** (Playwright, Cypress, Selenium, none?)
- **Coverage tool:** (c8, istanbul, coverage.py, coverlet, etc.)
- **Test data:** (factories, fixtures, Bogus/Faker, builders?)

## Step 2: Map Test Directory Structure

Document:
- Where unit tests live (next to source? separate directory?)
- Where integration tests live
- Where e2e tests live
- Where test utilities/helpers/fixtures live
- Where test configuration lives

## Step 3: Find the Best Test Example

Look for the test file with the most comprehensive patterns:
```
# Find test files with most lines (likely most thorough)
find . -type f \( -name "*.test.*" -o -name "*.spec.*" -o -name "*Test.cs" -o -name "*_test.go" \) -not -path '*/node_modules/*' -exec wc -l {} \; | sort -rn | head -10
```

Read the top 2-3 test files thoroughly — these are your reference test patterns.

## Step 4: Extract Test Conventions

### Test Naming
- What pattern? (`MethodName_Condition_Expected`? `describe/it`? `test_verb_noun`?)
- File naming? (`.test.ts`? `.spec.ts`? `Test.cs`? `_test.go`?)

### Test Structure
- Setup pattern (beforeEach? SetUp? arrange section?)
- Assertion style (expect? assert? Should?)
- Cleanup pattern (afterEach? Dispose? teardown?)
- AAA comments used? (Arrange/Act/Assert)

### Mocking Patterns
- How are dependencies mocked?
- How are API calls mocked? (MSW? nock? WireMock? manual?)
- Database mocking vs real database?

### Integration Test Patterns
- How is the application started for tests?
- Database setup (Testcontainers? in-memory? SQLite? test DB?)
- Authentication in tests (test tokens? bypass? fixtures?)
- Cleanup between tests

### Test Data
- How is test data created? (factories? builders? fixtures? inline?)
- Shared test data patterns?

## Step 5: Extract CI/Test Commands

Document the exact commands to:
- Run all tests
- Run unit tests only
- Run integration tests only
- Run with coverage
- Run a single test file
