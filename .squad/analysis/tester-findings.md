## 2. Coordinator ↔ Charter Alignment

### Issues Found

**[HIGH] Core coordinator hardcodes dotnet-angular cast names**

In `.github/agents/squad.agent.md:506-510`, the model selection table lists:
```
Lead/Architect (ripley)   — claude-opus-4.6
Backend (fenster)          — claude-sonnet-4.6
Frontend (dallas)          — claude-sonnet-4.6
Tester (hockney)           — claude-sonnet-4.6
```

And in `.squad/config.json:6`:
```json
"ripley": "claude-opus-4.6"
```

"ripley", "fenster", "dallas", "hockney" are all `dotnet-angular` cast names from `stacks/dotnet-angular/cast.conf`. These are **stack-specific names hardcoded into the core engine**. The core template's `team.md` only knows "Lead", "Backend", "Frontend", "Tester". A user installing core-only (or with a different stack) will:
- See unexplained names in the coordinator's model table
- Have a `"ripley"` model override in `config.json` that never matches any agent

**[MEDIUM] Routing table inconsistency between templates**

The `.squad/templates/routing.md` (stack template version) routes to `Ops` and `Docs` agents. The installed `.squad/routing.md` (core-only) routes docs to `Scribe` and has no `Ops`/`build`/`deploy`/`monitor` rows. The `routing.md` in `init.sh`'s inline heredoc (line 357–397) matches the simplified version. Result: the template and the installed file diverge — template is richer but that richness is lost in core installs.

**[LOW] Ceremonies.md minor divergence**

`.squad/ceremonies.md` (installed): Design Review triggers on "2+ agents modifying shared systems"  
`.squad/templates/ceremonies.md` (template): Design Review triggers on "2+ agents ... OR work touches 3+ layers"

The template is richer. The installed version loses the "3+ layers" trigger condition.

**[LOW] `agentModelOverrides` key resolution documented inconsistently**

Coordinator Layer 0.5 says: "If `agentModelOverrides.{agentName}` exists, use that model" — where `agentName` is the cast name (from `team.md`). But when agent names change via casting, old config.json keys become orphaned. No migration path is documented.

## 5. Config.json Issues

`.squad/config.json`:
```json
{
  "version": 1,
  "defaultModel": "claude-sonnet-4.6",
  "agentModelOverrides": {
    "lead": "claude-opus-4.6",
    "ripley": "claude-opus-4.6",
    "scribe": "claude-haiku-4.5",
    "ralph": "claude-haiku-4.5"
  }
}
```

**[HIGH] "ripley" in agentModelOverrides is a dotnet-angular cast name**

`team.md` defines the agent as "Lead" — the key `"lead"` correctly maps to it. But `"ripley"` (from `stacks/dotnet-angular/cast.conf: lead=ripley`) also appears. This means:
1. When dotnet-angular stack is installed, `init.sh` renames the agent directory from `lead/` to `ripley/` AND regenerates `team.md` with "Ripley" as the name. The `"ripley"` key in config.json then correctly applies. However...
2. The `"ripley"` key is in the **core** template's config.json — it will appear in every project that uses core-only install, dangling with no matching agent. The coordinator reads `agentModelOverrides.{agentName}` where agentName comes from `team.md` — "Lead" — so the "ripley" key is silently dead weight in core installs.
3. This is not a runtime error but signals poor separation between core and stack-specific config. The core config should not know about dotnet-angular cast names.

## 8. init.sh Edge Cases

### Issue A: [HIGH] Re-running init.sh on existing project silently overwrites charters

When `init.sh` is run WITHOUT `--upgrade` on a project that already has Squad installed, `init.sh:296`:
```bash
cp "$SCRIPT_DIR/core/.squad/agents/$agent/charter.md" "$TARGET/.squad/agents/$agent/charter.md"
```
This **unconditionally overwrites** all agent charters. No `-n` flag, no check for existing content. Compare with `config.json` at line 267 which uses `cp -n` (no-clobber). Users who have customized their charters (the primary use case for Squad) and then re-run init.sh will silently lose those customizations.

The `--upgrade` path correctly skips charters. But there is no warning or prompt in the fresh-install path: "Existing Squad detected. Use `--upgrade` to preserve customizations."

### Issue B: [MEDIUM] Seed-preset conflict silently resolved in favor of preset

When `--auto` detects matching seeds AND a matching preset exists (e.g., dotnet + angular → `dotnet-angular`), the script applies the preset which copies its `skills/*.md` over any seeds-based conventions. No warning is shown that seeds were superseded. This is fine for normal use but could confuse users who explicitly wanted just seeds behavior.

### Issue C: [LOW] --stack validation message mixes filtered and unfiltered output

At `init.sh:30`: `ls -1 "$SCRIPT_DIR/stacks/" | grep -v '^_' | grep -v 'seeds'`  
This correctly filters `_template` and `seeds` from the available presets list. ✅

At `init.sh:469-474`, the error path also does `ls -1 "$SCRIPT_DIR/stacks/" | grep -v '^_'` — this one does NOT filter `seeds`, so the error message "Available presets: ... seeds" would include `seeds` as if it were a valid preset. Minor but confusing.

### Issue D: [MEDIUM] No guard against running on a project that already has Squad (non-upgrade)

The script checks `[[ ! -d "$TARGET/.git" ]]` (must be a git repo) but does NOT check `[[ -d "$TARGET/.squad" ]]` to warn the user. A safe guard would be:
```bash
if [[ -d "$TARGET/.squad" && "$UPGRADE" == false ]]; then
  echo "⚠️  Existing Squad setup found. Use --upgrade to preserve customizations."
  echo "   Or continue to reset all charters (histories and decisions preserved)."
  read -r -p "Continue? [y/N] " confirm
  [[ "$confirm" =~ ^[Yy]$ ]] || exit 0
fi
```

### Issue E: [LOW] Detection of `dotnet` does not check subdirectory .csproj files

`init.sh:134`: `ls "$dir"/*.csproj >/dev/null 2>&1` only checks root-level `.csproj` files. Typical .NET solutions have `.csproj` files in subdirectories (`src/MyApp/MyApp.csproj`). A root-level `.sln` is checked separately which mitigates this, but a project with only `.csproj` in subdirectories won't be auto-detected as `dotnet`.
