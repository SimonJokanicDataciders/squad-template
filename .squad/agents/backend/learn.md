# Backend Agent — Codebase Learning Protocol

When instructed to learn the codebase, execute these steps IN ORDER.
Save your findings to `.copilot/skills/role-backend-core.md` using the skill bundle format.

---

## Step 1: Identify the Stack

Run these commands and read the results:

```
# Package manager / framework detection
ls package.json pyproject.toml Cargo.toml go.mod *.csproj *.sln Gemfile pom.xml build.gradle composer.json 2>/dev/null
```

Read the main package/project file to identify:
- **Language:** (TypeScript, Python, C#, Go, Rust, Java, etc.)
- **Framework:** (Express, FastAPI, ASP.NET, Gin, Actix, Spring, Laravel, etc.)
- **ORM/DB:** (Prisma, SQLAlchemy, EF Core, GORM, Diesel, Hibernate, Eloquent, etc.)
- **Test framework:** (Jest, pytest, xUnit, go test, etc.)

## Step 2: Map the Directory Structure

```
find . -type f -name "*.ts" -o -name "*.py" -o -name "*.cs" -o -name "*.go" -o -name "*.rs" -o -name "*.java" -o -name "*.php" -o -name "*.rb" | head -80
```

Identify and document:
- Where entities/models live
- Where services/business logic lives
- Where API routes/endpoints/controllers live
- Where DTOs/schemas/types live
- Where database migrations live
- Where configuration lives

## Step 3: Find the Reference Implementation

Look for the most complete feature — one that has:
- An entity/model
- A service or business logic layer
- API endpoints (CRUD)
- Tests

Search for the feature with the most files:
```
# Find feature-like directories with multiple file types
find . -type d -not -path '*/node_modules/*' -not -path '*/.git/*' -not -path '*/dist/*' -not -path '*/build/*' | while read dir; do
  count=$(find "$dir" -maxdepth 1 -type f | wc -l)
  [ "$count" -gt 3 ] && echo "$count $dir"
done | sort -rn | head -10
```

Read the top candidate thoroughly — this becomes the reference implementation.

## Step 4: Extract Code Conventions

Read 3-5 source files and document:

### Naming Conventions
- File naming (camelCase, PascalCase, kebab-case, snake_case?)
- Class/struct naming
- Function/method naming
- Variable naming
- Route/endpoint naming (plural? lowercase? prefix?)

### Code Patterns
- How are classes/structs declared? (sealed? public? exported?)
- How are DTOs/types defined? (interface? type? record? class?)
- How is dependency injection done? (constructor? decorator? function?)
- How are null/undefined checks done?
- How are errors handled? (try/catch? Result type? middleware?)
- How is validation done? (Zod? class-validator? FluentValidation? manual?)

### Entity/Model Pattern
Read a complete entity. Document:
- Base class or interface?
- Property style (init? readonly? public?)
- Relationships (how are FK references done?)
- Timestamps (created_at, updated_at?)
- Soft delete pattern?

### Service Pattern
Read a complete service. Document:
- Constructor / dependency injection style
- Method signatures (async? return types?)
- Database query patterns (raw SQL? ORM? compiled queries?)
- Transaction handling
- Error handling within services

### Endpoint/Route Pattern
Read a complete endpoint/controller. Document:
- How routes are registered
- How handlers are structured
- Request parsing (body, params, query)
- Response format (JSON shape, status codes)
- Middleware usage (auth, validation, etc.)
- Error response format

## Step 5: Extract Database Conventions

- How are migrations managed? (CLI command? auto-generate?)
- Connection setup pattern
- Query patterns (ORM methods, raw queries, compiled queries?)
- Seed data approach
