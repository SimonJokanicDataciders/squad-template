# CAP.Template — Known Squad Failure Patterns

This skill documents failure modes observed during Phase 1 Squad evaluation.
All agents should read this to understand accuracy risks and mitigation strategies.

## Observed Failure Modes

### 1. Hallucinated Method Names
**What happened:** Fenster + Hockney sync reported a `GetForecastAsync` public method on WeatherForecastService. This method does not exist.
**Mitigation:** Always cite exact method signatures with file:line references. Never report a method without reading the actual source file.

### 2. Wrong Access Modifiers
**What happened:** Sync output claimed `PostProcess` is public. It is actually private.
**Mitigation:** When reporting method visibility, quote the actual declaration line from source.

### 3. Wrong Return Values
**What happened:** For 25°C input, sync reported summary as "Mild". The actual value is "🌥️" (from WeatherSummaryProviderService).
**Mitigation:** Trace through the actual code path. Read the service, find the temperature-to-summary mapping, and verify the exact output.

### 4. Missing Method Coverage
**What happened:** Sync output failed to identify `GetForecasts` and both `UpdateForecast` overloads as methods needing test coverage.
**Mitigation:** Use grep/search to find ALL public and internal methods before claiming coverage assessment is complete.

### 5. Coordinator Model Mismatch
**What happened:** Despite .squad/config.json setting defaultModel to gpt-4.1, the coordinator ran as claude-sonnet-4.6.
**Mitigation:** The coordinator model is controlled by Copilot runtime, not Squad config. Only spawned agents can use gpt-4.1 via explicit task tool parameter.

### 6. Entity Property Without Migration (Phase 3)
**What happened:** Fenster added `DeletedAt` property to `WeatherData.cs` but did not create an EF Core migration. Hockney's integration tests all failed with "Invalid column name 'DeletedAt'" because the test database schema was out of sync.
**Mitigation:** When adding ANY new property to an entity, immediately run `dotnet ef migrations add {Name} --project src/Paso.Cap.Domain --startup-project src/Paso.Cap.Web`. Verify the migration file was created before proceeding.

### 7. Endpoint Handler Without Route Mapping (Phase 3)
**What happened:** Fenster created a `DeleteWeatherForecast` handler method in the endpoint class but forgot to register it in the `Map()` method with `routeBuilder.MapDelete(...)`. The endpoint was invisible at runtime.
**Mitigation:** After creating any endpoint handler, immediately verify it has a corresponding `routeBuilder.Map{Method}(...)` call in `Map()`. Every handler must be mapped.

### 8. Manually Editing Migration Snapshot (Phase 3)
**What happened:** Fenster edited `ApplicationDbContextModelSnapshot.cs` directly, removing the `#if PostgreSQL` conditional compilation and changing the ProductVersion. This would have broken dual-DB support permanently.
**Mitigation:** NEVER manually edit files in `Migrations/` or `*ModelSnapshot.cs`. These are auto-generated by EF Core. Only use `dotnet ef migrations add` to generate changes.

### 9. Unnecessary Package References (Phase 3)
**What happened:** Fenster added `Microsoft.EntityFrameworkCore.Relational` to the csproj, but this package was already available transitively through the SQL Server/Npgsql providers.
**Mitigation:** Only add NuGet packages when `dotnet build` actually fails with a missing type or namespace error.

### 10. Parallel Spawn of Dependent Agents (Phase 3)
**What happened:** Coordinator spawned Dallas (frontend) in parallel with Fenster (backend). Dallas needed the DELETE endpoint that Fenster was creating, but couldn't see it. Dallas reported "endpoint doesn't exist yet" and did no useful work.
**Mitigation:** Always check file dependencies before parallel spawn. If Agent B needs files Agent A will create, run them sequentially. Inline completed results as facts in the dependent agent's prompt.

## Mitigation Checklist (All Agents)

Before finalizing any code analysis output:
1. [ ] Every method name cited exists in the actual source (verified by search)
2. [ ] Every access modifier matches the actual declaration
3. [ ] Every return value was traced through the actual code path
4. [ ] Every file path cited exists and was read
5. [ ] Every line number referenced matches the actual file content
6. [ ] No method was "summarized" without reading its full implementation

### 11. Concurrent EF Core Queries on Same DbContext

**What happened:** Backend agent used `Task.WhenAll()` to run multiple EF queries in parallel, but all queries shared the same scoped `ApplicationDbContext`. EF Core's `DbContext` is NOT thread-safe — this caused `InvalidOperationException` at runtime (500 errors on the endpoint).

**Root cause:** The agent tried to optimize by parallelizing queries, but didn't know that DbContext is scoped per-request and must not be shared across threads.

**Mitigation:** NEVER run parallel EF Core queries on the same DbContext instance. Either:
- Run queries sequentially (simplest, correct for most cases)
- Inject `IDbContextFactory<T>` and create a separate context per parallel task

**WRONG:**
```csharp
var logins = _context.LoginHistory.Where(...).ToListAsync();
var actions = _context.RecentActions.Where(...).ToListAsync();
await Task.WhenAll(logins, actions); // CRASH — same DbContext
```

**CORRECT:**
```csharp
var logins = await _context.LoginHistory.Where(...).ToListAsync();
var actions = await _context.RecentActions.Where(...).ToListAsync();
// Sequential — safe, simple, correct
```

**Checklist:**
- [ ] No `Task.WhenAll` with multiple EF queries on the same context
- [ ] No `.AsParallel()` on IQueryable
- [ ] If parallelism is truly needed, use `IDbContextFactory<T>`
