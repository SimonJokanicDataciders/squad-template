## Guardrails

### Code Conventions (Embedded)
- **Sealed by default:** All classes `sealed` unless abstract base designed for inheritance
- **Immutable DTOs:** `sealed record` types with `init` properties or positional parameters; `IReadOnlyList<T>` for collections
- **Null checks:** `is null` / `is not null` only (NEVER `== null`)
- **XML docs:** Required on all public APIs, with `<example>` blocks when applicable
- **File-scoped namespaces**, K&R braces, `nameof()`, pattern matching, early returns
- **No speculative abstractions:** No interfaces for single-implementation services

### Entity Conventions
- `IAggregateRoot` marker interface
- `init` properties (except business-required mutations)
- `RowVersion` property always present
- Separate `IEntityTypeConfiguration<T>` file for FluentAPI

### Endpoint Conventions
- Extend `EndpointGroupBase`, call `.WithDefaultMetadata()` on every route
- Plural lowercase resource names under `/api` group (`/api/orders`)
- Route constraints (`{id:guid}`)
- Typed results: `Created` for POST, `NoContent` for PATCH/DELETE
- `JsonPatchDocument<T>` for partial updates
- `CancellationToken` on all async methods, services via `[FromServices]`
- **When adding a new endpoint handler, ALWAYS register the route in `Map()`** ‚Äî every `private` handler method MUST have a corresponding `routeBuilder.Map{Method}(...)` call in `Map()`. An unregistered handler is invisible at runtime.

### EF Core Conventions
- Compiled queries (`EF.CompileAsyncQuery`) or inline `.Select(x => new Dto(...))` for reads ‚Äî NEVER method groups or helper methods in `IQueryable.Select()`
- `ExecutionStrategy` wrapping all write operations
- Catch `DbUpdateConcurrencyException` as `ConcurrencyException`
- Dual-DB GUID: `Guid.NewGuid()` for MSSQL (non-clustered ID), `Guid.CreateVersion7()` for PostgreSQL (clustered, ordered)
- Migration `Down()` must have proper rollback logic ‚Äî never empty
- **NEVER edit files in `Migrations/` or `*ModelSnapshot.cs` manually** ‚Äî these are auto-generated by EF Core. Manual edits WILL break dual-DB (`#if PostgreSQL`) support. Use `dotnet ef migrations add` to generate migrations.
- When adding a new property to an entity, you MUST create a migration: `dotnet ef migrations add {Name} --project src/Paso.Cap.Domain --startup-project src/Paso.Cap.Web`
- Never use `FromSqlRaw` with string interpolation
- **NEVER run parallel EF Core queries on the same DbContext** ‚Äî `DbContext` is NOT thread-safe. Use sequential `await` calls. If parallelism is truly needed, inject `IDbContextFactory<T>` and create a separate context per task.

### Package References
- Do NOT add NuGet package references unless `dotnet build` fails with a missing type/namespace error
- EF Core provider packages (`Microsoft.EntityFrameworkCore.SqlServer`, `Npgsql.EntityFrameworkCore.PostgreSQL`) already include `Microsoft.EntityFrameworkCore.Relational` transitively ‚Äî never add it manually

### Testing Conventions
- xUnit, `MethodName_Condition_ExpectedResult` naming
- No AAA comments
- Run tests before committing
